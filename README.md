# نظام إدارة العملاء — UIS

منصة متكاملة بشريط جانبي موحّد، بـ 4 واجهات: **الإدارة**، **المستخدمون**، **السيلز**، **التيكنيكال سبورت**، شغالة على قاعدة بيانات واحدة (Supabase) ومستضافة مجانًا على Vercel.

بيانات مشروعك على Supabase متحطة بالفعل جوه `assets/app.js`.

## هيكل المشروع
```
index.html            → بيوجّه أوتوماتيك لـ login.html
login.html            → تسجيل الدخول (بدون شريط جانبي)
sales.html             → عملائي: Kanban + نموذج اكتشاف + بيانات مالية + عقد + جهات اتصال + مرفقات + متابعات
support.html           → مراجعة كل النماذج + Debrief + مرحلة التنفيذ
management.html        → لوحة KPIs + Charts + تارجت + تجديدات + مصادر عملاء + نشاط الفريق
users.html              → إدارة المستخدمين والأدوار + سجل التدقيق (admin بس)
report.html             → تقرير التنفيذ والـ Go-Live (بدون شريط جانبي، صفحة تفصيلية مستقلة)
assets/styles.css        → التصميم + الشريط الجانبي
assets/app.js            → الاتصال بـ Supabase + دوال مشتركة + renderSidebar
supabase/functions/create-user/index.ts → Edge Function لإنشاء مستخدمين من الموقع
schema.sql .. schema_v8.sql → كل تحديثات قاعدة البيانات بالترتيب
```

## خطوات التشغيل من الصفر (رتّبها بالظبط كده)

### ١. قاعدة البيانات
في **SQL Editor**، شغّل الملفات **بالترتيب**: `schema.sql` → `schema_v2.sql` → `schema_v3.sql` → `schema_v5.sql` → `schema_v6.sql` → `schema_v7.sql` → `schema_v8.sql` → `schema_v9.sql`
(تجاهل `schema_v4.sql` — كانت فكرة تذاكر دعم مش محتاجينها، وده الملف الوحيد اللي تتجاهله).

لو عندك مستخدمين اتعملوا قبل تحديث `schema_v8.sql`، شغّل هذا السطر مرة واحدة عشان يتملأ عمود الإيميل بتاعهم:
```sql
update public.profiles p set email = u.email from auth.users u where p.id = u.id and p.email is null;
```

### ٢. خطوة واحدة في لوحة Supabase (مرة واحدة بس)
عشان الحسابات الجديدة تتفعّل فورًا من غير أي إيميل تأكيد:
1. روح لمشروعك في supabase.com → **Authentication → Providers → Email**
2. شيل علامة الصح من **"Confirm email"**
3. احفظ

### ٣. إنشاء أول حساب Admin (يدويًا، مرة واحدة بس)
لسه محتاج تعمل **أول** حساب admin يدويًا من **Authentication → Users → Add user**، وتغيّر دوره لـ admin من **Table Editor → profiles**. بعد كده، أي مستخدم جديد ينفع يتعمل من صفحة `users.html` مباشرة.

### ٤. الرفع على Vercel
ارفع مجلد المشروع كامل (بما فيه `assets/`) على vercel.com.

## المميزات الجديدة في التحديث ده

### إدارة المستخدمين من الموقع (`users.html`)
- جدول بكل المستخدمين مع Dropdown لتغيير الدور فورًا
- زرار "+ إضافة مستخدم": بيطلب الاسم، الإيميل، كلمة مرور، والدور، وبينشئ الحساب **فورًا وفعّال من غير أي إيميل تأكيد** (طالما خطوة "Confirm email" في الإعدادات متعملة)
- سجل تدقيق (Audit Log) بيوضح آخر 30 تغيير حصل في مراحل العملاء وحالات المراجعة وأدوار المستخدمين، ومين عمل التغيير

### شريط جانبي موحّد
كل الصفحات (ما عدا تسجيل الدخول وتقرير Go-Live) بقت بشريط جانبي ثابت، والتابات بتختلف حسب دورك تلقائيًا.

### تطوير نموذج السيلز
- **جهات اتصال إضافية**: تقدر تضيف أكتر من شخص (صاحب قرار، مسؤول مالي...) لكل عميل، مع رابط واتساب مباشر لكل رقم
- **تاريخ المتابعة القادمة**: لو فات وميتسجّلش نشاط بعده، الكارت في الـ Kanban بيوريك badge "متابعة متأخرة"
- **سبب الخسارة**: بيتسجّل مع تحديث المرحلة لـ"خسارة"، ومفيد لتحليل الخسائر بعدين
- **مرفقات**: رفع أي ملف (عرض سعر، عقد ممضي) لكل عميل، وتحميله لاحقًا بلينك آمن مؤقت

## الحماية
كل الصلاحيات محكومة من قاعدة البيانات (Row Level Security)، بما فيها المرفقات (Storage).

**نقطة مهمة تستحق الوضوح**: إنشاء المستخدمين بقى بيتم بالكامل من كود الموقع (بدون Edge Function ولا سيرفر إضافي)، عن طريق نفس مفتاح `anon key` العام. عشان كده الحسابات بتتفعّل فورًا من غير تأكيد إيميل — لازم تشيل "Confirm email" من إعدادات Supabase (موضح فوق). **العيب الوحيد**: بما إن التسجيل شغال بمفتاح عام، أي حد يعرف رابط مشروعك التقني (مش الموقع نفسه) نظريًا يقدر يسجّل حساب "sales" افتراضي بنفسه لو حاول مباشرة عن طريق الـ API (مش هيقدر يشوف أي بيانات لحد ما admin يفعّل صلاحيته، لكنه هيقدر يعمل حساب فاضي). ده تحت مستوى مخاطرة مقبول لفريق داخلي صغير، لكن لو حابب تقفلها تمامًا لاحقًا، الحل هو الرجوع لأسلوب الـ Edge Function.

## بيانات ديمو (لعرض المنصة على الشركة)

عندك ملف `demo_seed_data.sql` بيضيف 9 عملاء واقعيين موزّعين على كل المراحل والدول (مصر/الكويت/عُمان)، مع Debriefs، أنشطة، تقارير تنفيذ، وأهداف شهرية — عشان تقدر تعمل عرض كامل من غير ما تدخل بيانات يدويًا.

**شرط قبل التشغيل**: لازم يكون عندك مستخدم واحد على الأقل بدور "sales" في جدول profiles (اعمله من `users.html` أو Authentication). شغّل `demo_seed_data.sql` بعد كل ملفات schema.

**لمسح بيانات الديمو بعد العرض**: استخدم `reset_test_data.sql` الموجود بالفعل — بيمسح كل البيانات ويسيب حسابات المستخدمين زي ما هي.

## تحديث مهم: الكود بقى مدموج جوه كل صفحة (v16)

بعد محاولات متكررة لتشخيص مشكلة تحميل، اكتشفنا إن المشكلة على الأغلب كانت في إزاي Vercel بيبعت ملف `assets/app.js` كملف منفصل (نوع الملف Content-Type). عشان نقفل الاحتمال ده تمامًا، **كود `app.js` بقى مدموج مباشرة جوه كل صفحة HTML** بدل ما يكون ملف خارجي منفصل. ده معناه:

- **مفيش حاجة تتغير في طريقة الاستخدام** — كل صفحة شغالة بنفس الطريقة بالظبط
- **مفيش اعتماد على تحميل ملف JS خارجي من نفس السيرفر** — القابلية للمشكلة دي اتقفلت خالص
- ملف `assets/app.js` لسه موجود في المشروع كمرجع، بس مش مستخدم فعليًا من أي صفحة دلوقتي
- `assets/styles.css` لسه ملف خارجي عادي (التنسيق مش بيأثر على وظائف الموقع لو فشل تحميله، أسوأ حالة الصفحة تبان بلا تنسيق، مش "الموقع واقف")

## تحديث: رجوع لملف JS خارجي + إعدادات قابلة للتعديل (v19)

- **رجعنا `assets/app.js` يبقى ملف خارجي عادي** (بعد ما اتأكدنا إن `vercel.json` بيجبر نوع الملف (Content-Type) الصحيح، السبب الأصلي للمشكلة اتقفل من غير الحاجة للدمج جوه كل صفحة). ده بيسهّل الصيانة جدًا — أي تعديل بقى في ملف واحد بس.
- **أسعار الصرف واحتماليات الإغلاق بقت قابلة للتعديل من الموقع نفسه**، مش أرقام مكتوبة في الكود. شغّل `schema_v10.sql` بعد `schema_v9.sql`، وهتلاقي في صفحة `users.html` قسمين جداد:
  - **أسعار الصرف**: عدّل السعر وقت ما يتغيّر، بيتحفظ فورًا ويتطبق على كل الحسابات
  - **احتمالية الإغلاق لكل مرحلة**: بجانب كل نسبة مفترضة، هتلاقي عمود "المعدل الفعلي من بياناتكم" — نسبة العملاء اللي فعلاً وصلوا للمرحلة دي وكسبوا الصفقة، محسوبة من بيانات الشركة الحقيقية. استخدمها كمرجع تقرر بيه هل تعدّل الافتراض ولا لأ.

## تحديث: رجّعنا للكود المدمج (Inline) بشكل نهائي — مع سكريبت بناء تلقائي

جرّبنا نرجّع `assets/app.js` ملف خارجي (مع `vercel.json` لضبط نوع الملف)، لكن المشكلة استمرت. رجعنا لأسلوب الدمج (Inline) لأنه مضمون 100%، لكن عشان الصيانة متبقاش صعبة، فيه ملف جديد: **`build.py`**.

### طريقة الشغل من دلوقتي
1. عدّل في `assets/app.js` **بس** — نفس الملف، مفيش حاجة تتغيّر في عادتك
2. شغّل من الترمينال (لو عندك Python 3):
   ```
   python3 build.py
   ```
3. السكريبت هيدمج نسخة `app.js` المحدّثة جوه كل صفحة تلقائيًا، ويزوّد رقم النسخة لوحده
4. ارفع الملفات المحدّثة على Vercel

لو معندكش Python متاح، ابعتلي التعديل اللي عايزه في `assets/app.js` وأنا هعمل الدمج وأبعتلك الملفات جاهزة زي ما بنعمل دايمًا.
